# Local Codex Workflow For This Router Project

Цель: работать дешевле, спокойнее и надежнее. Не каждый раз заново разбираться в хаосе, а запускать понятный процесс.

## Главное правило

Ты задаешь цель и границу риска. Я выбираю способ: сам делаю, зову младшего агента на сбор фактов, запускаю sanitizer, или прошу подтверждение перед опасным действием.

## Что писать мне в начале задачи

Хороший формат:

```text
Цель: чтобы ChatGPT mobile работал через роутер.
Симптом: country not available.
Можно трогать роутер: да/нет.
Можно пушить в GitHub: да/нет.
Успех: приложение открылось на телефоне.
```

Не обязательно всегда так формально. Но эти пять пунктов резко экономят лимиты и нервы.

## Когда подключать младших агентов

Младшим агентам хорошо отдавать:

- поиск связанных файлов;
- разбор логов;
- сверку README и конфигов;
- поиск доменов/IP, которые идут `direct`;
- проверку, не попали ли секреты в публичный repo;
- подготовку чернового отчета.

Не стоит отдавать младшим агентам:

- финальное решение, что публиковать;
- применение конфигов на роутер;
- расширение VPN-правил "на всякий случай";
- действия с сетью Mac, которые что-то меняют.

## Встроенные project-skills

Этот repo содержит проектовые навыки:

```text
.codex/project-skills/keenetic-repo-triage/
.codex/project-skills/xray-routing-triage/
.codex/project-skills/public-config-sanitizer/
.codex/project-skills/macos-vpn-diagnostician/
```

Их смысл:

- `keenetic-repo-triage` - быстро понять, какие файлы затронуты;
- `xray-routing-triage` - разобрать симптомы и логи Xray;
- `public-config-sanitizer` - проверить, что в public не уехали ключи;
- `macos-vpn-diagnostician` - объяснить сетевые симптомы Mac человеческим языком.

## Definition of Done

Для роутерных изменений задача считается законченной, когда есть:

- backup или понятный rollback;
- проверенный JSON;
- `xkeen -status` без аварии;
- пользовательское подтверждение, что целевой сервис работает;
- публичный repo обновлен только sanitized-файлами.

## Директорский взгляд

Слепая зона здесь была не техническая, а процессная: мы долго тушили отдельные пожары. Правильная система такая:

1. Логируем симптом.
2. Собираем факты дешево.
3. Меняем минимально.
4. Проверяем на целевом устройстве.
5. Фиксируем знание в repo.

Так проект превращается из разовой настройки в воспроизводимый актив.
